# Phase 1: 对齐功能边界

**目标：** 产出「功能边界简报」——明确源仓、目标仓、功能范围、约束与验收标准。

## 步骤

### 1.1 收集必要信息

按顺序确认，**已答项跳过**：

1. **源仓库**：路径或名称；功能在源仓中的叫法/入口
2. **目标仓库**：路径或名称；是否已 clone / 可访问
3. **功能边界**：具体做什么、不做什么（用户故事或验收标准）
4. **语言/框架差异**：同语言还是跨语言？目标仓技术栈？
5. **适配约束**：必须遵守的目标仓约定（目录结构、命名、状态管理等）
6. **验收标准**：怎样算「功能对等」？

若源仓或目标仓路径已知，先 `move_agent_to_root` 到对应工作区再收集细节。

### 1.2 输出功能边界简报

```markdown
## 功能边界简报

**功能名：** …
**源仓库：** `{path}` — 功能入口/关键词：…
**目标仓库：** `{path}` — 技术栈：…
**跨语言/跨框架：** 是 / 否 — 说明：…

**In scope（本任务必须覆盖）：**
- …

**Out of scope（明确不做）：**
- …

**验收标准（P0）：**
- [ ] …

**目标仓适配约束：**
- …

### 待确认
- [ ] 以上是否准确？
```

## 门禁

用户明确确认（「可以」「确认」「继续」）后 → Phase 2。

**禁止：** 未确认前读取源仓代码或开始 GitNexus 探索。
