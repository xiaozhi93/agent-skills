# Phase 1: 功能选择与技术栈

**目标：** 产出「MVP 意图简报」——对齐保留哪些功能、用什么技术栈、新项目放哪里。

## 步骤

### 1.1 快速扫描原仓（5 分钟内）

不深入读代码，仅识别：

- 应用类型（移动 / Web / 后端 / 全栈）
- 主要模块（从目录名、README、包结构推断）
- 原技术栈（语言、框架、构建工具）

### 1.2 列出功能模块清单

按用户可理解的功能（非代码模块）列出，例如：

```markdown
## 原应用功能模块

| # | 模块 | 简述 | 建议 |
|---|------|------|------|
| 1 | 用户登录 | 账号密码 + OAuth | 核心 |
| 2 | 商品列表 | 分页浏览 | 核心 |
| 3 | 支付 | 第三方支付 SDK | 可选 |
| 4 | 推送通知 | FCM/APNs | 可砍 |
| … | … | … | … |
```

**建议列**帮助用户决策，但最终以用户勾选为准。

### 1.3 技术栈决策

**一次一问**，确认：

1. **沿用原栈还是换栈？**
   - 沿用：记录原栈版本与构建方式
   - 换栈：让用户说明目标栈及理由（学习成本、跨平台、团队熟悉度）
2. **新项目路径？** 默认 `~/Projects/<app-name>-mvp/` 或用户指定
3. **非功能约束？** 如最低系统版本、离线支持、数据库选型

### 1.4 输出 MVP 意图简报

```markdown
## MVP 意图简报

**原项目：** /path/to/source
**保留功能：** 1, 2, …
**排除功能：** 3, 4, …
**技术栈：** Kotlin + Jetpack Compose（沿用）/ React Native（换栈）
**新项目路径：** ~/Projects/myapp-mvp/
**约束：** …

### 待确认
- [ ] 功能清单与技术栈是否准确？
```

## 门禁

用户明确确认（「可以」「确认」「继续」）后 → Phase 2。
