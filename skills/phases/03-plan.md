# Phase 3: MVP 规划

**目标：** 产出「MVP 实现计划」——功能规格、项目结构、实现顺序，供用户确认后进入开发。

## 步骤

### 3.1 裁剪到 MVP 边界

对每个保留功能，明确：

- **Must have：** MVP 必须交付的最小行为
- **Won't have：** 本阶段明确不做（防止范围蔓延）

示例：

```markdown
### 用户登录
- Must: 邮箱+密码登录，保存 session/token
- Won't: OAuth、忘记密码、生物识别
```

### 3.2 设计新项目结构

按目标技术栈给出目录骨架，例如：

```
myapp-mvp/
├── src/
├── tests/          # 仅核心路径
├── docs/
│   └── MVP-SPEC.md
└── README.md
```

### 3.3 实现顺序

按依赖关系排序，通常：

1. 项目 scaffold + 构建验证
2. 共享基础设施（网络、存储、路由）
3. 核心功能路径（按用户价值排序）
4. 最小 UI 串联

### 3.4 输出 MVP 实现计划

```markdown
## MVP 实现计划

**项目路径：** ~/Projects/myapp-mvp/
**技术栈：** …

### 功能规格
（各功能的 Must / Won't）

### 项目结构
（目录树）

### 实现顺序
1. …
2. …

### 风险与假设
- …

### 待确认
- [ ] 计划是否批准？
```

## 可选：生成交接提示词

若 MVP 较大、需新会话继续，可用 `generate-dev-handoff-prompt` 生成交接提示词。

## 门禁

用户明确批准计划后 → Phase 4。
