# Generate Dev Handoff Prompt

将「需求文档 + 代码分析结论 + 用户补充要求」合成为**可直接粘贴到新对话**的开发提示词。本技能只负责**生成提示词**，不在当前会话开始写代码。

## Overview

**核心原则：** 分析结论 → 边界对齐 → 自包含提示词 → 自检交付。新会话 agent 无需翻阅旧对话即可开工。

**产出：**

1. 一句话交付说明（含建议配套技能）
2. 单个可复制的 Markdown 开发提示词（fenced code block）
3. 可选：`docs/handoff/YYYY-MM-DD-<feature-slug>.md`（仅当用户要求保存）

**前置条件（生成前确认已具备）：**

1. **需求文档**（原文或用户确认的摘要）
2. **代码分析**（相关文件路径、现有模式、影响范围、风险点）
3. **用户补充**（优先级、取舍、必须/禁止项）

若缺任一项，先向用户索取或基于对话上下文补全，并标注 `[待确认]`。

## When to Use

- 需求分析会话已完成，用户要「开新对话开发」
- 用户说「生成开发提示词」「整理成 prompt」「/export dev prompt」
- 已有完整分析，只需合成 handoff 给实现 agent
- `repo-feature-distill` Phase 3 之后需要给目标仓开发会话的提示词

**When NOT to use:**

- 尚未完成需求或代码分析 → 先完成分析再生成
- 用户要在当前会话直接写代码 → 用 `writing-plans` 或常规开发流程
- 资源散落、需 Phase 0 归集 → 用 `feature-phase0-kickoff`
- 只改一两处小需求 → 直接描述任务，不必走完整 handoff 模板

## Red Flags

| 失误 | 后果 |
|------|------|
| 缺代码分析就生成提示词 | 文件路径臆造，新会话无从下手 |
| 生成提示词的同时开始写代码 | scope 混乱，handoff 未完成 |
| 留 `{待填写}` 等空占位符 | 新会话 agent 阻塞或自行猜测 |
| 未写 Out of scope | 新会话发散实现无关功能 |
| P0 验收标准不可判定 | 无法确认完成与否 |
| 提示词含 API key、密码 | 安全风险 |
| 分析过程全文塞入提示词 | token 浪费，关键决策被淹没 |

**STOP 信号：**

- 「差不多能开发了，先写点代码吧」→ 先完成自检清单
- 「路径你看着办」→ 回到代码分析补全文件清单
- 「不用写 Out of scope」→ 必须写明边界

## 工作流

### Step 1：收集与对齐

从对话中提取并去重：

| 来源 | 提取内容 |
|------|----------|
| 需求文档 | 目标、用户故事、验收标准、非功能需求 |
| 代码分析 | 入口文件、依赖模块、可复用模式、需改动的文件清单 |
| 用户补充 | 覆盖需求文档的决策、技术选型、范围裁剪 |

**冲突处理**：用户补充 > 本轮分析建议 > 需求文档默认表述。在提示词中显式写出已采纳的冲突决议。

### Step 2：划定实现边界

明确写入提示词：

- **In scope**：本对话必须完成的能力
- **Out of scope**：明确不做的事项（防止新会话发散）
- **Open questions**：仅保留阻塞实现的未决项（≤3 条）；非阻塞项给出默认假设

### Step 3：生成开发提示词

使用下方模板输出**一个完整 Markdown 代码块**，用户复制后在新对话首条消息发送。

### Step 4：自检

生成后对照检查清单（见文末）。未通过则修订后再输出。

## 输出格式

向用户交付：

1. **一句话说明**：提示词用途与建议配套技能（如有）
2. **可复制的开发提示词**（单个 fenced code block）
3. **可选**：将同内容写入 `docs/handoff/YYYY-MM-DD-<feature-slug>.md`（仅当用户要求保存）

**禁止**：在生成提示词的同时开始实现代码（除非用户明确要求在当前会话开发）。

## 开发提示词模板

将 `{...}` 替换为实际内容；删除不适用的章节，不要留空占位符。

```markdown
# 开发任务：{功能名称}

> 本提示词由需求分析会话生成。请按以下说明实现，不要重新做需求调研，除非标注为 [待确认]。

## 目标

{一句话描述要交付什么、为谁、解决什么问题}

## 背景与上下文

{2-4 句：业务背景、当前系统行为、为何现在要做}

## 需求摘要

### 必须实现（P0）

- {验收标准 1 — 可测试、可判定完成}
- {验收标准 2}

### 应当实现（P1）

- {…}

### 明确不做（Out of scope）

- {…}

## 用户决策与约束（优先于需求文档）

以下为用户在分析阶段确认的要求，**必须遵守**：

1. {用户补充 1}
2. {用户补充 2}

## 代码库上下文（分析结论）

### 相关文件与模块

| 路径 | 作用 | 本次操作 |
|------|------|----------|
| `{path}` | {说明} | 修改 / 新增 / 只读参考 |

### 应遵循的现有模式

- {命名、目录结构、状态管理、API 风格等，引用具体文件}

### 影响与风险

- {破坏性变更、迁移、兼容性、性能等}

## 实现方案（已确认）

{3-6 句：推荐架构/数据流/关键接口；若有多方案，写清为何选此方案}

### 关键步骤

1. {步骤 — 含具体文件路径}
2. {步骤}
3. {步骤}

## 测试与验收

### 手动验证

- [ ] {场景 1}
- [ ] {场景 2}

### 自动化（如适用）

- 运行：`{具体命令}`
- 预期：{通过条件}

## 工作方式

- 先阅读 `{核心入口文件}` 再动手
- 每完成一个逻辑块执行验证，不要累积大批量未测改动
- 遵循项目既有风格；不要引入未讨论的新依赖
- {项目特定约束，如：不提交、不 push、不改 xxx}

## [待确认]（仅阻塞项）

- {问题} — 默认假设：{假设}（若用户未回复则按此执行）

## 完成定义（Definition of Done）

- [ ] 所有 P0 验收标准满足
- [ ] 测试/验证步骤已执行并通过
- [ ] 无 linter 错误引入（如项目有检查）
- [ ] {其他项目级 DoD}
```

## 质量要求

- **自包含**：新会话 agent 无需翻阅旧对话即可开工
- **可执行**：文件路径、命令、验收标准必须具体，禁止「适当处理」「按需实现」
- **可判定**：每条 P0 验收标准能明确判断完成与否
- **精简**：去掉分析过程的推理过程，只保留结论与决策
- **中文**：用户用中文交流时，提示词正文用简体中文（代码/路径/命令保持原文）

## 建议搭配的实现技能

在交付说明中，按任务类型推荐（不写入提示词正文，除非用户要求）：

| 场景 | 建议技能 |
|------|----------|
| 多步骤、需拆任务 | `writing-plans` → `executing-plans` |
| 需先澄清设计 | `brainstorming` |
| 改前评估影响 | `gitnexus-impact-analysis` |
| TDD 开发 | `test-driven-development` |

## 自检清单

- [ ] P0 验收标准均可测试
- [ ] 用户补充已体现且优先级正确
- [ ] 文件路径来自真实代码分析，非臆造
- [ ] Out of scope 已写明
- [ ] 无 API key、密码等敏感信息
- [ ] 无 `{待填写}` 类空占位符
- [ ] 输出为单个可复制代码块

## 触发示例

用户说：

- 「根据上面的分析和我的要求，生成开发提示词」
- 「我要开新对话开发，帮我整理成 prompt」
- 「/export dev prompt」

即启用本技能。
