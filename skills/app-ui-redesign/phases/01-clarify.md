# Phase 1: 澄清重设计范围

**目标：** 产出「重设计意图简报」——对齐重设计哪些功能/屏幕、参考设计、平台栈、分支策略。

**前置：** 读取 `brainstorming` 技能，做创意工作前对齐意图。

## 步骤

### 1.1 快速扫描原仓（5 分钟内）

不深入读代码，仅识别：

- 应用类型（Web / iOS / Android / React Native / 跨平台）
- 主要屏幕/页面（从路由、导航、目录推断）
- 原技术栈与 UI 框架（保持原栈，不换技术栈）
- 是否已有 `DESIGN.md` 或设计 token 文件

### 1.2 确认重设计范围

**一次一问**，按顺序：

1. **整 App 还是部分功能？**
   - 整 App → Phase 3 必须产出完整 `DESIGN.md`
   - 部分 → 列出具体屏幕/模块，用户勾选
2. **参考设计？** 截图、链接、品牌规范、竞品、现有 Stitch 项目 ID
3. **风格方向？** 如深色/浅色、极简/活泼；无参考时 Phase 3 可用 `taste-design`
4. **非功能约束？** 无障碍、RTL、最低系统版本、必须保留的交互

### 1.3 确认实施约束

- **原仓路径** 与 **feature 分支名**（默认 `ui-redesign/<short-name>`）
- 可选：是否用 `using-git-worktrees` 隔离工作区
- Stitch 项目：新建 vs 沿用已有（若有 project ID 记录）

### 1.4 输出重设计意图简报

```markdown
## 重设计意图简报

**原项目：** /path/to/repo
**范围：** 整 App / 部分（列出屏幕）
**保留交互：** …
**参考设计：** …
**平台栈：** React Native 0.73（沿用）
**分支：** ui-redesign/home-login
**Stitch 项目：** 新建 / projects/xxx
**整 App → 需 DESIGN.md：** 是 / 否

### 待确认
- [ ] 以上是否准确？
```

## 门禁

用户明确确认（「可以」「确认」「继续」）后 → Phase 2。
